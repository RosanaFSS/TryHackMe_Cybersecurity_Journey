<h2>Task 1 . Introduction to Identity Management<a id='1'></a></h2>
<h3>What Is Privileged Identity Management (PIM)?<a id='1.1'></a></h3>
<p>Microsoft Entra <strong>Privileged Identity Management (PIM)</strong> is a service that enables the <strong>discovery</strong>, <strong>control</strong>, and <strong>monitoring</strong> of <strong>privileged assignments</strong> in Microsoft Entra, Azure, and other Microsoft services such as Microsoft 365 and Microsoft Intune. PIM allows configuring <strong>just-in-time</strong> access and setting up relevant settings for privileged roles so that users will have to activate and be approved for relevant roles every time they need to use the role.</p>

<h6 align="center"><img width="800px" src="https://github.com/user-attachments/assets/6104de1e-bcb4-4ea7-ab69-10c137b6397a"><br>This image and all the theoretical content of the present article is TryHackMe´s property.</h6>

<h3>Why Use PIM?<a id='1.2'></a></h3>
<p>Using <strong>Privileged Identity Management</strong> (which will be called PIM for the rest of this room) helps organizations reduce their <strong>Identity Attack Surface</strong>strong> by managing and discovering privileged roles and monitoring excessive resource permissions. By doing so, organizations reduce the risk of a malicious actor getting access and an authorized user inadvertently impacting a sensitive resource.</p>

<h3>What Are the Key Features of PIM?<a id='1.3'></a></h3>
<p>Microsoft Entra PIM has several key features that help achieve Zero Trust and Least Privilege best practices.</p>

<h6 align="center"><img width="800px" src="https://github.com/user-attachments/assets/a97622b3-5110-4c7c-a176-2122dfc2f179"><br>This image and all the theoretical content of the present article is TryHackMe´s property.</h6>

<p>Using the above features, <a href="https://learn.microsoft.com/en-us/entra/identity/role-based-access-control/permissions-reference#privileged-role-administrator">Privileged Role Administrators</a> can be on top of the requested and standing privileged access on the directory and Azure resources.</p>

<h3>Who Can Use PIM?<a id='1.4'></a></h3>
<p>In Microsoft Entra, only <strong>Privileged Role Administrator (PRA)</strong>strong> or <strong>Global Administrator</strong>strong> roles can <strong>manage</strong>strong> assignments for other administrators. On the other hand, all the below roles can <strong>view</strong>strong> the assignments in PIM:<br>

- <a href="https://learn.microsoft.com/en-us/entra/identity/role-based-access-control/permissions-reference#global-administrator">Global Administrator</a><br>
- <a href="https://learn.microsoft.com/en-us/entra/identity/role-based-access-control/permissions-reference#security-administrator">Security Administrator</a><br>
- <a href="https://learn.microsoft.com/en-us/entra/identity/role-based-access-control/permissions-reference#global-reader">Global Reader</a><br>
- <a href="https://learn.microsoft.com/en-us/entra/identity/role-based-access-control/permissions-reference#global-reader">Security Reader</a><br>

Now that we have scratched the surface of PIM, we will next see some of the terminology and the concept of the four A's: <strong>Assign</strong>, <strong>Activate</strong>, <strong>Approve</strong>, and <strong>Audit</strong>.</p>

<h2>Task 2 . Privileged Identity Management: Four A´s<a id='2'></a></h2>
<p>Different types of users, such as administrators and members, will interact with PIM differently. That's why, before we discuss how things work in PIM, we need to understand certain PIM-specific concepts around role assignments. Let's start with the <strong>Four A's</strong>.</p>

<h6 align="center"><img width="800px" src="https://github.com/user-attachments/assets/a1e88a32-6202-4d98-9b7e-a08c28c3cefc"><br>This image and all the theoretical content of the present article is TryHackMe´s property.</h6>

<h3>Assign<a id='2.1'></h3>
<p>This is simply creating user assignments for Microsoft Entra roles to ensure that users have Just-in-Time (JIT) access to privileged roles. Let's try to elaborate on this with some questions:<br>

- Who can assign privileged roles to other users?<br><strong>Privileged Role Administrator (PRA)</strong> or <strong>Global Administrator</strong>.<br>
- To whom are these privileged roles assigned?<br>Regular users who need to perform privileged tasks in a JIT manner.<br>
- How long will the users have privileged roles assigned to themselves?<br>For a limited time, such as 2hrs/4hrs/8hrs, defined by the administrator.</p>

<h3>Activate<a id='2.2'></h3>
<p>Following an assignment, certain users know that as part of their job, they may need to perform some privileged tasks occasionally. For instance:<br>

- A <strong>Developer Lead</strong> may need to manage applications registered in the directory for the apps the developer team is creating. Hence, an <strong>Application Administrator</strong> role will be required to perform these tasks for a limited time.<br>
- Similarly, a <strong>Human Resources (HR)</strong> admin may need to manage some users and groups. In this case, the HR admin will elevate access to the User Administrator role using PIM, again for a limited time.<br>

The idea is that users <strong>eligible</strong> for certain privileged roles can <strong>activate</strong> their roles in a self-service fashion. We will discuss the eligibility concept further. Administrators can also directly activate the eligible roles if needed.</p>

<h3>Approve<a id='2.3'></h3>
<p>With PIM, roles can be configured to require <strong>approval for activation, and one or multiple users or groups can be chosen</strong> as <strong>delegated approvers</strong>. Delegated approvers have 24 hours to approve requests. If a request is not approved within 24 hours, the eligible user must re-submit a new request. The 24-hour approval time window is not configurable.</p>

<h3>Audit<a id='2.4'></h3>
<p>Audit history can be used to see all role assignments and activations within the past 30 days for all privileged roles. It will list all the details on:<br>

- Who requested the privileged assignment?<br>
- Who granted the privileged assignment?<br>
- What was the action taken?<br>
- What was the primary target, i.e., the privileged role assigned?</p>

<h3>Assignment Types<a id='2.5'></h3>

<h6 align="center"><img width="400px" src="https://github.com/user-attachments/assets/0aa7c327-4a3a-48ce-8dc3-55466be33c90"><br>This image and all the theoretical content of the present article is TryHackMe´s property.</h6>

<p>In Microsoft Entra PIM, mainly there are two types of assignments:<br>

- <strong>Eligible</strong>: The user is eligible for the assigned privileged role. However, the user needs to activate the role to perform privileged tasks.<br>For example, an HR admin is eligible for the User Administrator role. After activating it, the HR admin gets the User Administrator role.<br>
- <strong>Active</strong>: The user's privileged role assignment is directly activated. i.e., no need for further activation.<br>

In addition, start and end dates can be set for the above assignment types, which gives the following four assignment types in total:<br>

- Eligible (Permanent)<br>
- Active (Active)<br>
- Time-bound (Eligible)<br>
- Time-bound (Active)</p>
